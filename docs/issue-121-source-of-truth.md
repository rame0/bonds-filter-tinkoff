# Issue #121: единый источник данных по облигациям

## Что сделано в этой ветке

- кеш backend переведен с файлового `file-system-cache` на SQLite
- поиск, фильтрация, сортировка и пагинация перенесены на сервер
- UI больше не загружает весь список облигаций целиком

Это закрывает дополнительный комментарий в issue про хранение и server-side filtering.

## Какие поля реально нужны приложению

По коду UI и backend сейчас используются такие группы полей:

- идентификация и листинг: `uid`, `figi`, `isin`, `ticker`, `name`
- торговые и брокерские признаки: `buyAvailableFlag`, `sellAvailableFlag`, `apiTradeAvailableFlag`, `forIisFlag`, `forQualInvestorFlag`, `otcFlag`, `weekendFlag`
- профиль инструмента: `currency`, `countryOfRisk`, `sector`, `issueKind`, `realExchange`, `riskLevel`, `couponQuantityPerYear`
- ценовые поля: `price`, `nominal`, `aciValue`, `realPrice`
- доходность и дюрация: `bondYield`, `couponsYield`, `duration`
- купоны и погашение: `coupons`, `leftCouponCount`, `leftToPay`, `buyBackDate`, `maturityDate`, `floatingCouponFlag`, `amortizationFlag`, `perpetualFlag`, `subordinatedFlag`
- ликвидность: `liquidity`

## Матрица покрытия

| Поле / группа | MOEX | Tinkoff | Комментарий |
| --- | --- | --- | --- |
| Список доступных у брокера облигаций | частично | да | Tinkoff остается source of truth по universe для этого приложения |
| `uid`, `figi`, `isin`, `ticker`, `name` | частично | да | MOEX не закрывает брокерские идентификаторы полностью |
| Цена (`price`) | нет | да | в приложении берется из `getLastPrices` Tinkoff |
| Номинал, НКД, валютные поля | частично | да | Tinkoff уже отдает их в нормализованном виде |
| График купонов (`coupons`) | да | да | но Tinkoff подвержен rate limit, MOEX удобнее для массовой загрузки |
| Сумма купонов за год (`couponsYield`) | да | нет | в Tinkoff это считается вручную из купонов |
| Доходность (`bondYield`) | да | нет | сейчас это MOEX-поле, с fallback-расчетом только для части бумаг |
| Дюрация | да | нет | без MOEX сейчас остается только грубый расчет по дате погашения |
| Ликвидность | да | нет | текущая логика строится только на истории торгов MOEX |
| Риск, сектор, тип выпуска, страна | нет | да | это метаданные Tinkoff |
| Брокерские флаги доступности | нет | да | MOEX не знает, доступна ли бумага в Tinkoff и на каких условиях |

## Что ломается в вариантах single-source

### MOEX-only

Плюсы:

- есть купоны, доходность, дюрация и ликвидность

Минусы:

- теряется broker-specific universe Tinkoff
- теряются `uid` и часть идентификаторов, нужных для портфеля и запросов купонов
- теряются торговые и регуляторные флаги Tinkoff
- теряются `riskLevel`, `sector`, `issueKind`, `countryOfRisk`, `realExchange`

Примеры облигаций, которые уже сейчас выпадают из MOEX-only:

- `XS1589106910` / `Credit Bank of Moscow`
- `XS0191754729` / `Gazprom`
- `RU000A107PM2` / `Мегатакт-НН БО-01`
- `OMA0148` / `Sultanate of Oman`
- `SIBN6P3` / `Газпром Нефть 006Р-03`

### Tinkoff-only

Плюсы:

- не теряется список доступных у брокера бумаг
- сохраняются все пользовательские и брокерские признаки

Минусы:

- нет готового поля ликвидности
- нет биржевой доходности и нормальной дюрации как системных полей
- массовая загрузка купонных графиков идет через отдельные запросы и упирается в rate limit
- придется заново рассчитывать часть полей, которые MOEX уже отдает готовыми

Примеры бумаг, где MOEX сейчас нужен для рыночных полей:

- `RU000A0ZYX28` / `Совкомбанк 1В02` (`BondYield` отсутствует в MOEX-выгрузке, для таких бумаг уже нужен fallback)
- `SU29022RMFS9` / `ОФЗ 29022` (рыночные поля требуют дообработки)
- `RU000A1022E6` / `ПР-Лизинг БО 002Р-01` (`couponsYield` в MOEX не готов, сейчас добирается через fallback)

Важно:

- по списку инструментов Tinkoff-only ничего не теряет
- по качеству полей Tinkoff-only все равно не покрывает текущие требования без дополнительного слоя расчетов

## Рекомендация

Рекомендуемый вариант: **явно задокументированная гибридная схема**.

Роли источников стоит зафиксировать так:

- Tinkoff: source of truth для списка доступных у брокера облигаций и всех broker-specific метаданных
- MOEX: source of truth для рыночных полей `bondYield`, `duration`, `liquidity`, массового чтения купонов и расчетного `couponsYield`
- fallback: только как явное исключение для бумаг, где MOEX не отдал `bondYield` или `couponsYield`

## Почему не стоит переключаться на один источник прямо сейчас

- MOEX-only отрезает часть бумаг, уже доступных пользователю в Tinkoff
- Tinkoff-only сохраняет universe, но ломает текущую аналитику по доходности, ликвидности и производительности массовой загрузки купонов
- текущий practical win находится не в single-source, а в упрощении хранения и выдачи данных

## Следующий технический шаг после этой ветки

- вынести роли источников в отдельный `SOURCE_OF_TRUTH.md` или раздел `README`
- в модели данных явно хранить provenance для вычисляемых полей: `bondYieldSource`, `couponsYieldSource`, `durationSource`, `liquiditySource`
- логировать список тикеров, где сработал fallback Tinkoff, чтобы видеть реальную деградацию MOEX по времени
